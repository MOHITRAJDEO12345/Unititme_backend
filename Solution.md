Master Solution Architecture: Eduniketan Allocation EngineTarget Deployment: Lovely Professional University (LPU)Scale: 40,000+ Students, High-Density Campus GeographyTech Stack: Node.js, Supabase (PostgreSQL), React/Vue, GCP (Dockerized UniTime)1. Executive SummaryThe Eduniketan Allocation Engine is a decoupled, headless scheduling architecture designed to solve the NP-Hard Student Sectioning and Master Timetabling problem for massive academic institutions.Instead of building a heuristic solver from scratch, Eduniketan utilizes an AI-assisted Node.js middleware to bridge a modern LMS frontend (Supabase/React) with a world-class open-source constraint solver (UniTime). This allows LPU administrators to manage complex constraints through a modern, intuitive UI, while offloading the heavy mathematical computation to a dedicated optimization engine.2. System Architecture (The 3-Tier Model)The system is strictly divided into three isolated layers:Tier 1: The System of Record & UI (Eduniketan LMS)Database: Self-hosted Supabase (PostgreSQL). Holds all raw entity data (Students, Rooms, Faculty, Courses) and the final master_timetable.Frontend: React/Vue dashboard where LPU administrators ingest data, view the "Data Health" dashboard, and adjust "Constraint Sliders" (e.g., setting the penalty for Day Scholar gaps).End-User Display: Students and Faculty view their personalized schedules by querying the Supabase database directly. UniTime is completely invisible to the end-user.Tier 2: The Middleware Bridge (Node.js & Antigravity)Data Translator: Node.js microservices that pull relational data from Supabase and compile it into UniTime’s required DataExchange XML format.Constraint Compiler: Converts user-friendly UI slider values (e.g., "Strictly Forbid") into UniTime’s mathematical severity tags (Prohibited, Strongly Discouraged).State Manager: Triggers the Batch Solver API, polls for completion, and parses the resulting XML/JSON back into relational Supabase tables.Tier 3: The Optimization Engine (UniTime Server)Infrastructure: GCP e2-standard-8 (8 vCPU, 32GB RAM) running Docker Compose.Components: Runs a localized, zero-latency PostgreSQL container purely for the solver's internal scratchpad, alongside the UniTime Java JVM (allocated 26GB Heap).Lifecycle: Ephemeral processing. It wakes up, receives XML payloads, crunches the schedule for 1-4 hours, returns the optimized solution, and goes dormant.3. Data Ontology & Ingestion PipelineTo satisfy LPU's physical and academic realities, the data is mapped from Supabase to XML via the following ontology:Infrastructure (The Campus Map): * Rooms are tagged with <roomFeature> tags (e.g., BYOD_READY).The "LPU Sprint" (Zero-Buffer Transit) is handled via a generated traveltimes.xml payload. Connected blocks are assigned 0 minutes travel time; disconnected blocks are assigned high penalty minutes.Offerings (The Academics): * Courses are expanded dynamically based on expected_students. A 600-student course generates ten 60-capacity <class> sections dynamically in Node.js.Theory/Lab durations are mapped to minutesPerWeek in the <schedulingSubpart>.Students (The Load): * 40,000 students are mapped with <studentMajors> and <studentMinors> to guarantee Pathway integrity.Students with BYOD requirements are injected into a <studentGroup> to ensure they are only placed in BYOD_READY mapped rooms.4. Constraint Management LifecycleAdministrators do not interact with XML. Constraints are managed via a complete abstraction layer:UI Definition: The Eduniketan UI features a "Rule Builder" (e.g., Toggle: Prevent back-to-back classes in different blocks).Node.js Compilation: Upon execution, Node.js compiles these rules into UniTime <distributionPrefs>.Severity Mapping Matrix:UI "Mandatory" $\rightarrow$ XML Pref="Prohibited" (Hard Constraint / Category 1)UI "High Priority" $\rightarrow$ XML Pref="Strongly Discouraged" (Operational / Category 2)UI "Low Priority" $\rightarrow$ XML Pref="Discouraged" (Experience / Category 3)Injection: These preferences are attached to the Course Offerings XML payload before hitting the solver.5. The Execution Loop (The "Batch Solve")When the LPU Administrator clicks "Generate Master Timetable":Pre-Flight Check: Node.js runs the "Data Health" check (flagging missing minors, room capacity mismatches).Payload Generation: Node.js builds students.xml, offerings.xml, rooms.xml, and traveltimes.xml.Push to Engine: Node.js POSTs all XML files to http://[GCP_VM_IP]:8888/unitime/api/exchange.Solver Trigger: Node.js POSTs to the /api/solver endpoint to initialize the Batch Student Sectioning Solver.Polling: Node.js polls the engine every 60 seconds checking the job status.Extraction & Persistence: Once the solver converges, Node.js calls the Solution Export API, parses the master JSON, and performs a bulk INSERT into the Supabase student_sections and master_timetable tables.Publishing: The Eduniketan frontend updates to reflect the newly populated Supabase tables via standard SQL queries/GraphQL.6. System Boundaries & ScalingSolver Constraint: The solver's maximum capacity is limited by JVM Heap Memory. The 32GB RAM allocation on GCP is strictly designed to hold the massive constraint matrix of 40k students simultaneously.LMS Scaling: Because the final schedule is extracted as flat relational data back to Supabase, the Eduniketan frontend can scale infinitely using Supabase's native CDN and read-replicas without putting any load on the UniTime solver.
